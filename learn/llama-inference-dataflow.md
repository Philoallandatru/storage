## 四、关键代码路径（llama.cpp）

```cpp
// 1. KV Cache 创建（初始化阶段）
llama_kv_cache::llama_kv_cache(...) {
    // 为每一层创建 K 和 V 张量
    for (uint32_t il = 0; il < n_layer; il++) {
        ggml_tensor * k = ggml_new_tensor_3d(
            ctx, type_k, 
            n_embd_k_gqa,  // K 的维度
            kv_size,        // 最大序列长度
            n_stream        // 并行流数量
        );
        
        ggml_tensor * v = ggml_new_tensor_3d(
            ctx, type_v, 
            n_embd_v_gqa, 
            kv_size, 
            n_stream
        );
        
        layers.push_back({ il, k, v, ... });
    }
}

// 2. 写入 KV Cache（推理时）
ggml_tensor * llama_kv_cache::cpy_k(
    ggml_context * ctx, 
    ggml_tensor * k_cur,     // 新计算的 K [n_tokens, n_embd_k]
    ggml_tensor * k_idxs,    // 写入位置索引
    int32_t il,              // 层索引
    const slot_info & sinfo  // slot 信息
) const {
    // 从 cache 获取目标位置
    ggml_tensor * k_cache = layers[il].k;
    
    // 使用 ggml_set_rows 操作将新的 K 写入指定位置
    // k_cache[k_idxs] = k_cur
    return ggml_set_rows(ctx, k_cache, k_cur, k_idxs);
}

// 3. 读取 KV Cache（推理时）
ggml_tensor * llama_kv_cache::get_k(
    ggml_context * ctx, 
    int32_t il,         // 层索引
    uint32_t n_kv,      // 要读取的 KV 数量
    const slot_info & sinfo
) const {
    // 返回该层的 K cache 视图
    ggml_tensor * k_cache = layers[il].k;
    
    // 创建一个视图，只包含有效的 KV
    return ggml_view_2d(ctx, k_cache, 
                        n_embd_k_gqa, 
                        n_kv,  // 只读取前 n_kv 个位置
                        ...);
}

// 4. Attention 计算（使用 KV Cache）
// 在 llama_build_graph() 中：
for (int il = 0; il < n_layer; il++) {
    // 计算新 token 的 Q, K, V
    ggml_tensor * Q = ggml_mul_mat(ctx, W_q, cur);
    ggml_tensor * K_new = ggml_mul_mat(ctx, W_k, cur);
    ggml_tensor * V_new = ggml_mul_mat(ctx, W_v, cur);
    
    // 写入 KV cache
    K_new = kv_cache->cpy_k(ctx, K_new, k_idxs, il, sinfo);
    V_new = kv_cache->cpy_v(ctx, V_new, v_idxs, il, sinfo);
    
    // 从 cache 读取完整的 K, V（包括历史）
    ggml_tensor * K_full = kv_cache->get_k(ctx, il, n_kv, sinfo);
    ggml_tensor * V_full = kv_cache->get_v(ctx, il, n_kv, sinfo);
    
    // Attention 计算
    ggml_tensor * scores = ggml_mul_mat(ctx, K_full, Q);  // [n_kv, 1]
    scores = ggml_scale(ctx, scores, 1.0 / sqrt(d_k));
    scores = ggml_soft_max(ctx, scores);
    ggml_tensor * output = ggml_mul_mat(ctx, V_full, scores);
}
```

## 五、性能优化要点

### 1. 为什么 KV Cache 能加速？

```
不使用 KV Cache：
  Token 1: 计算 Q1, K1, V1 → attention(Q1, K1, V1)
  Token 2: 重新计算 Q1, K1, V1, Q2, K2, V2 → attention(Q2, [K1,K2], [V1,V2])
  Token 3: 重新计算 Q1..Q3, K1..K3, V1..V3 → attention(Q3, [K1,K2,K3], [V1,V2,V3])
  
  计算复杂度：O(n^2) 对于生成 n 个 token

使用 KV Cache：
  Token 1: 计算并缓存 K1, V1
  Token 2: 只计算 Q2, K2, V2，从 cache 读取 K1, V1
  Token 3: 只计算 Q3, K3, V3，从 cache 读取 K1, K2, V1, V2
  
  计算复杂度：O(n) 对于生成 n 个 token
  
加速比：生成 100 个 token，快约 50 倍！
```

### 2. 内存与计算的权衡

```
优势：
  ✅ 大幅减少重复计算
  ✅ 加速 token 生成（典型快 10-100 倍）
  ✅ 可以处理长上下文

代价：
  ❌ 占用大量内存（长上下文 = 几 GB）
  ❌ 内存带宽成为瓶颈
  ❌ 限制批处理大小
```

### 3. llama.cpp 的优化技术

```
1. 量化 KV Cache:
   - FP16 → INT8: 减少一半内存
   - FP16 → INT4: 减少 75% 内存
   
2. GQA (Grouped Query Attention):
   - 共享 K, V heads
   - 70B 模型：8 个 KV heads vs 64 个 Q heads
   - 减少 KV cache 8 倍
   
3. 分层存储:
   - 热层在 GPU VRAM
   - 冷层在 DRAM
   - 极限情况：NVMe SSD (手动保存/加载)
   
4. Flash Attention:
   - 分块计算 attention
   - 减少 HBM 访问
   - 不减少 KV cache 大小，但加速计算
```

## 六、总结

**核心要点：**

1. **Prefill 阶段**：处理输入 prompt，生成并存储所有位置的 KV
2. **Decode 阶段**：每次只处理 1 个新 token，读取历史 KV + 追加新 KV
3. **KV Cache 结构**：`[n_layers, 2, kv_size, n_embd]`
4. **性能关键**：避免重复计算历史 token 的 K 和 V
5. **内存瓶颈**：长上下文需要数 GB 的 KV cache

**实际应用建议：**

- 短对话（<2K tokens）：KV cache 在 VRAM 中即可
- 长对话（2K-32K tokens）：考虑混合存储（部分在 DRAM）
- 超长上下文（>32K tokens）：需要会话管理 + 状态保存
