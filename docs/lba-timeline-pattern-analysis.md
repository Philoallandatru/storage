# LBA Timeline 图表模式分析

**日期:** 2026-07-14  
**数据源:** KV Cache I/O 三路对比测试（ShareGPT、BurstGPT、Default）  
**分析工具:** bpftrace tracepoint:block:block_rq_issue + plot_lba_rw_timeline.py  
**目的:** 深度解读 LBA timeline 时间序列图的访问模式，理解 KV cache offload 的真实 I/O 特征

---

## 1. 为什么 LBA Timeline 图开始时有蓝色读操作？

### 1.1 现象观察

在 LBA timeline 散点图（第 4 子图）中，**时间轴的最开始（0-5s）就出现了蓝色散点（读操作）**，而不是预期中的先写后读模式。

**疑问：**
- 读的内容是什么？
- Cache 还没建立，为什么会有读？
- 为什么不是先写入 KV cache，再读取？

### 1.2 读操作的来源

**这些初始读操作主要来自以下几个方面：**

#### **来源 1：模型权重加载（最主要）**

```
测试配置：
- GPU/CPU KV cache: 0/0 GiB（强制使用 NVMe）
- 模型：llama3.1-8b
- 测试开始时需要加载模型权重
```

**读 I/O 特征：**
- 模型权重文件通常数 GB（llama3.1-8b 约 8-16 GB）
- 以 **128 KiB 块大小**读取（与 KV cache block 大小一致）
- 相对**顺序或局部连续**读取（权重文件是连续存储的）
- 发生在测试的**最开始几秒**

**证据：**
从三路对比报告的数据可以看到：
- LBA span = 389.38 GiB（整个测试涉及的 LBA 范围）
- 但总传输数据量远小于 LBA span
- 说明有大范围的 LBA 被访问，但不是全部写入

#### **来源 2：文件系统元数据读取**

```
文件系统操作序列：
1. 打开 KV cache 目录
2. 读取目录元数据（inode、dentry）
3. 检查已有的 cache 文件（如果有）
4. 准备创建新文件
```

**I/O 特征：**
- 小块随机读（通常 4K-64K）
- 分散在不同 LBA 位置
- 数量相对较少，但在时间轴上**最早出现**

#### **来源 3：KV Cache 文件的预分配/初始化检查**

某些文件系统（如 ext4）在创建大文件时：
1. 先读取 block bitmap 和 inode bitmap
2. 分配 extent（连续块）
3. 可能触发 journal 读取
4. 这些都是**读操作**，发生在**写入实际数据之前**

#### **来源 4：Cache 预热/已有数据检查**

如果测试环境：
- **没有清空 cache 目录**
- 上一轮测试遗留了 KV cache 文件
- Runner 会先**读取已有文件**来判断是否可复用

**注意：** 三路对比测试使用的配置是强制使用 NVMe offload，但**没有明确说明是否清空了 cache 目录**。

### 1.3 为什么不是"先写后读"？

**预期的逻辑（错误假设）：**
```
测试开始
  → Prefill 新请求
  → 写入 KV cache 到 SSD
  → 后续 Decode 读取 KV cache
  → 应该先看到红点（写），再看到蓝点（读）
```

**实际的逻辑（正确理解）：**
```
测试开始
  → 加载模型权重（读 I/O）✅
  → 读取文件系统元数据（读 I/O）✅
  → Prefill 新请求
  → 写入 KV cache 到 SSD（写 I/O）
  → Decode 阶段读取 KV cache（读 I/O）
```

**关键点：**
- **模型加载的读 I/O 先于 KV cache 写入**
- **bpftrace 采集的是块设备层所有 I/O**，不区分来源
- LBA timeline 图是**块设备视角**，包含模型加载、文件系统、KV cache 的所有 I/O

## 2. LBA Timeline 图的完整时间演进

### 2.1 第一阶段：系统初始化（0-5s）

**I/O 特征：**
- ✅ **蓝色散点（读）先出现**
- ✅ 相对**聚集在某些 LBA 区域**
- ✅ 读占比可能较高（50-70%）
- ✅ IOPS 相对较低（系统刚启动）

**主要操作：**
1. 模型权重加载（数 GB 读 I/O）
2. 文件系统元数据读取
3. Cache 目录初始化
4. Python 运行时加载

### 2.2 第二阶段：冷启动期（5-20s）

**I/O 特征：**
- ✅ **红色散点（写）开始密集出现**
- ✅ **写占比上升**（可能达到 40-60%）
- ✅ LBA 访问范围开始扩大

**主要操作：**
1. Prefill 新请求
2. 写入 KV cache blocks 到 SSD
3. Cache 池逐步建立

### 2.3 第三阶段：过渡期（20-60s）

**I/O 特征：**
- ✅ **蓝色散点逐渐增多**（KV cache 读开始主导）
- ✅ **读占比上升**到 70-90%
- ✅ LBA 访问范围覆盖更广

**主要操作：**
1. Cache hit rate 逐步提升
2. Decode 阶段开始主导
3. 读取之前写入的 KV cache blocks

### 2.4 第四阶段：稳态期（60s+）

**I/O 特征：**
- ✅ **蓝色散点占据绝对主导**（覆盖整个 Y 轴）
- ✅ **读占比达到 90%+**
- ✅ LBA 访问模式稳定

**主要操作：**
1. Cache hit rate 达到稳态（BurstGPT ~98%, ShareGPT ~72%）
2. Decode 完全主导
3. 持续从整个 cache 池中随机读取 KV blocks

## 3. 如何区分"模型加载读"和"KV Cache 读"

### 3.1 基于时间特征

| 特征 | 模型加载读 | KV Cache 读 |
|---|---|---|
| **出现时间** | 0-5s（最早） | 10s+（逐渐增多） |
| **持续时间** | 短暂（几秒） | 持续整个测试 |
| **密度变化** | 初期高，快速下降 | 初期低，逐渐上升 |

### 3.2 基于空间特征

| 特征 | 模型加载读 | KV Cache 读 |
|---|---|---|
| **LBA 分布** | 相对连续或局部聚集 | 全 LBA 空间随机分布 |
| **LBA 跳跃** | 小跳跃或连续 | 大跨度跳跃（≥100 MiB） |
| **覆盖范围** | 局部区域 | 整个 cache 池 |

### 3.3 基于读占比曲线

观察**第 3 子图：每秒读占比**：

**模型加载主导期（0-5s）：**
- 读占比可能**暂时较高**（50-80%）
- 然后**快速下降**（写入开始）

**KV Cache 主导期（60s+）：**
- 读占比**稳定在高位**（90%+）
- 不会再大幅波动

**如果只看散点图（第 4 子图）容易误解，必须结合读占比曲线（第 3 子图）判断。**

## 4. 对 AI SSD 设计的启示

### 4.1 不要忽视模型加载的读 I/O

**现实场景：**
- 模型加载是服务启动的**第一步**
- 数 GB 甚至数十 GB 的读 I/O
- 如果 SSD 读延迟高，**服务启动慢**

**设计要求：**
- ✅ 大块顺序读优化（模型权重加载）
- ✅ 大块随机读优化（KV cache 读取）
- ✅ 两者都重要，不能只优化一种

### 4.2 读写模式在时间维度上演进

**启动阶段（0-20s）：**
- 读（模型加载）+ 写（cache 建立）
- 需要**读写并发性能**

**稳态阶段（60s+）：**
- 读主导（90%+）
- 需要**读 tail latency 优化**

**设计要求：**
- ✅ 不能只看稳态，启动阶段也很重要
- ✅ QoS 需要动态适应读写比例变化

---

## 5. 核心结论

### 5.1 关于初始读操作

✅ **LBA timeline 图开始时的蓝色读操作主要来自：**
1. **模型权重加载**（最主要，数 GB）
2. 文件系统元数据读取
3. Cache 文件的预分配检查
4. 可能的已有 cache 检查

✅ **不是 KV cache 读取**
- KV cache 读取在 10s+ 后才逐渐成为主导
- 稳态期（60s+）才是真正的 KV cache 随机读

### 5.2 关于"先写后读"的误解

❌ **错误理解：** 测试开始应该先写入 cache，再读取  
✅ **正确理解：** 块设备视角包含所有 I/O，模型加载的读先于 cache 写入

### 5.3 关于图表解读

✅ **必须结合 4 个子图综合判断**
- IOPS 曲线：看趋势
- 带宽曲线：看量级
- 读占比曲线：看演进（关键！）
- LBA 散点图：看空间模式

### 5.4 读写比例演进（估算）

| 阶段 | 时间范围 | 读占比 | 主要 I/O 来源 |
|---|---|---:|---|
| 系统初始化 | 0-5s | 60-80% | 模型加载读 |
| 冷启动期 | 5-20s | 30-50% | Cache 写入 |
| 过渡期 | 20-60s | 70-85% | Cache 读写混合 |
| 稳态期 | 60s+ | 90%+ | Cache 随机读 |
