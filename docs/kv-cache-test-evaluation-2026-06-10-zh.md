# KV 缓存跨厂商测试 — 关键评估 (KV Cache Cross-Vendor Test — Critical Evaluation)

**日期 (Date)**: 2026-06-10
**范围 (Scope)**: 对 `docs/kv-cache-cross-vendor-2026-06-10.md` 的方法论审查
**结论 (Verdict)**: 数据内部一致，每盘排名稳健，但若干参数选择意味着**该测试并未衡量真实生产环境 LLM 推理 KV 缓存流量**。核心结论（Biwin 胜出）是正确的；但绝对数值不应被引用为生产环境的预测。

---

## P0 — 关键方法论问题 (Critical methodological issues)

### 问题 1: `--gpu-mem-gb 0 --cpu-mem-gb 0` 跳过了分层级联

**我们做了什么**: 完全禁用了 GPU 和 CPU 层，强制 100% 的读写操作落在 NVMe 上。

**为什么有问题**: 在实际的 LLM 推理中，KV 缓存瀑布是 **GPU HBM → CPU DRAM → NVMe**，只有*溢出部分*才会触及 NVMe。通过禁用上层，我们将 NVMe 视为*唯一*的存储层来测量。实际生产中的 NVMe 流量会是：

- **数量更少** — 可能只有总缓存触及次数的 5–20%（其余命中 HBM 或 DRAM）
- **偏差更大** — 旧对话的长尾读取、大提示词预填充期间的大顺序写入
- **不同的 I/O 形状** — 更少的小随机读取，更多预填充期间的大顺序写入

**影响**: 我们的读取 P99 数值（Biwin K1 约 14 ms）可能*优于*生产环境（因为我们总是进行冷读取，没有内存缓存），而吞吐量数值（Biwin K4 的 7,157 tok/s）可能*差于*生产环境（因为我们没有 HBM 来吸收热路径）。

**修复**: 设置 `--gpu-mem-gb 80 --cpu-mem-gb 80`，匹配单 H100 推理箱的 80 GB HBM + 80 GB DRAM 层级。重新运行 K2 和 K4。

### 问题 2: `--trace-speedup 1000` 将 33 小时的轨迹压缩到 120 秒

**我们做了什么**: 以 1000 倍墙钟速度回放 BurstGPT 轨迹。

**为什么有问题**: 在 1000 倍速度下，每次 120 秒的运行代表约 33 小时的现实用户流量。这意味着：

- 每块盘被要求在 2 分钟内处理约 33 小时的请求
- 每"墙钟秒"的请求速率比实际生产高 1000 倍
- 轨迹的*自然*请求间隔被破坏 — 突发模式被保留但到达间隔时间被压缩

在这种机制下，我们看到的 98% 缓存命中率是可疑的 — 在真实时间里，相同的轨迹会有更低的命中率，因为缓存驱逐压力有时间累积。

**影响**: KV 缓存是"热的"，因为所有数据都适合无限制的缓存目录，且轨迹回放速度足够快，没有条目会老化退出。在实际生产中，有界缓存目录加上现实时间，命中率会更低，瓶颈会转向*驱逐策略*而非存储吞吐量。

**修复**: 将 `--trace-speedup` 降至 **1.0–10×**。在 10× 时，每次 120 秒运行代表约 20 分钟的真实流量，这更接近实际。

### 问题 3: `--max-concurrent-allocs 2` 将并发度限制为 2

**我们做了什么**: 无论 `--num-users` 为何值，都将并发分配数设为 2。

**为什么有问题**: 有 16 个用户时，自然并发分配率应为约 16（每用户一个）。限制为 2 意味着：

- KV 缓存子系统排队请求
- 存储流量被人为节流
- 我们在*轻负载*队列深度下测量存储吞吐量
- 高 `--num-users` 测试变成了"16 个用户在队列大小为 2 中等待"，而非"16 个用户同时加载磁盘"

**影响**: 这**抑制了我们想要在高并发下对 NVMe 施加的实际压力**。WD 的 K4 P99 为 328 ms 已经很差 — 但如果我们让它真正以 16 个并发分配运行，结果可能更差（或更好，如果主机端队列进行了平滑处理）。

**修复**: 移除 `--max-concurrent-allocs 2` 上限，或将其设置为 `--num-users` 的值（允许完全并发）。

## P1 — 重要注意事项 (Important caveats)

### 问题 4: `--cache-dir` 没有大小限制

我们从未设置最大 KV 缓存大小，因此每次运行时 cache_dir 无限制增长。这意味着：

- 每次运行的总读取量达到 200–500 GB
- 所有磁盘活动舒适地适合缓存
- "未命中率"人为偏低，因为没有任何数据被驱逐

**实际生产环境**会有有界的 KV 缓存池（例如 100–500 GB），命中率将由驱逐策略与工作集大小的交互决定。我们没有测量这种交互。

**修复**: 使用 `--storage-capacity-gb 200` 来限制缓存。

### 问题 5: `--replay-cycles 0` — 仅单次回放

我们每次运行仅回放一次轨迹。在 `trace-speedup=1000` 下，这意味着运行*期间*的突发速率巨大，但每次运行的*总数据量*仅为轨迹的一份副本。要使 GC 反压起作用（T3/T4 显示 GC 是一个巨大因素），我们需要写入足够的总数据来填满 SLC 并深入到 TLC。我们在每次 Biwin K5 运行中仅写入约 40 GB，这**小于 Biwin 的 SLC 缓存**（T2 发现 >168 GB SLC）。

**修复**: 要么将 K5 运行更长时间（600 s = 5 分钟），要么使用 `--replay-cycles` 多次重复轨迹。

### 问题 6: 测试运行时间过短，不足以观察 GC 效应

先前跨厂商表征的 T3/T4 显示 GC 漂移需要约 5–15 分钟才会显现。我们的 K1–K4 运行时间为 120 秒（2 分钟）— 远远不足以看到 GC 反压。只有 K5（180 秒）开始接近 GC 起作用的区域。

**影响**: Biwin 的*突发*性能主导了结果，而 T3/T4 已经显示 Biwin 在 15 分钟 GC 压力下持续吞吐量下降 30%。**在现实的长时推理服务器场景中，Biwin 相对于 ZhiTai 的优势将大幅缩小。**

## P2 — 次要问题 (Minor issues)

### 问题 7: BurstGPT 轨迹是单一工作负载

我们只使用了 BurstGPT 轨迹（Azure OpenAI 聊天）。实际的 LLM 推理服务包括：
- 代码补全（长预填充，短解码）
- 文档问答（非常长的预填充，非常短的解码）
- 多轮对话（预填充 + 解码循环，前缀缓存密集）
- 批量作业（预填充密集，无解码）

BurstGPT 仅是聊天。代码/RAG 工作负载会有非常不同的存储流量形状（更多预填充写入，更少解码读取）。

**修复**: 添加 ShareGPT 轨迹工作负载（`fio_sharegpt_*.ini` 已在 `results/kvcache-profile/` 中）并重新运行 K2/K4。

### 问题 8: 无预热阶段

我们在 `--drop_caches` 后直接运行每个场景。实际生产服务器有已经预热数小时的热缓存。每次运行的前 10–20 秒比稳态更冷。

**修复**: 添加 30 秒预热，从摘要中排除其指标（`--enable-latency-tracing` 已通过轨迹时间戳支持这一点）。

### 问题 9: `direct=1` 是隐式的（我们未设置）

KV 缓存基准测试默认可能使用缓冲 I/O。在页缓存热的情况下，读取可能来自 DRAM 而非磁盘。结合无限制的 `--cache-dir`，这可能掩盖实际的 NVMe 负载。

**修复**: 强制存储层读取使用 O_DIRECT（需要检查 `kv-cache.py` 是否支持）。

### 问题 10: 单一种子 (Single seed)

我们只使用了 `--seed 42` 运行。给定轨迹 + 加速 + 种子，BurstGPT 轨迹回放是确定性的，因此这实际上没问题 — 但第二个种子可以捕捉到任何依赖种子的异常。

## 数据确实可靠地告诉我们什么 (What the data DOES tell us reliably)

尽管存在以上问题，以下结论是**稳健的**：

1. **Biwin X570 在高并发下具有最佳的小随机 R/W 存储控制器** — 在 K3/K4 中可见，读取 P99 为 48 ms / 60 ms。这个排名与 T5（QD=64 的 4K 随机 IOPS）一致，其中 Biwin 也是最佳的配备 DRAM 的盘。领先是真实的，不是测量伪影。

2. **ZhiTai 在 70B 下写入尾部延迟崩溃** — K5 中写入 P99 为 1,073 ms vs Biwin 的 28 ms，差距如此之大（38 倍），即使页缓存带来 5 倍的实际改善，ZhiTai 在写入尾部仍然比 Biwin 差约 7 倍。"避免 ZhiTai 用于大模型"的结论成立。

3. **WD SN570 在并发下饱和** — 从 K2 到 K4 的 3,500 tok/s 平台是真实的瓶颈，不是测量伪影。即使 HBM 层吸收了部分负载，无 DRAM 的 WD 仍然落后于配备 DRAM 的盘。

4. **Seagate FC530 是 Biwin 在 70B 下最强的替代方案** — 28 ms 写入 P99 与 Biwin 持平，2,012 tok/s 在 Biwin 的 2,521 tok/s 的 25% 以内。T6 混合 R/W 的领先优势转移到了 KV 缓存大模型。

5. **所有盘的缓存命中率约 98%** — 这是稳健的，因为它取决于*轨迹*而非磁盘。2–3% 的未命中路径才是磁盘差异化的所在，而尾部延迟差异正是出现在那里。

## 结论 (Bottom line)

**相对排名是正确的**：对于 KV 缓存工作负载，Biwin > Seagate ≥ ZhiTai > WD 是正确的判断。

**绝对数值是被夸大的**：tok/s 数值可能*低于*生产环境（无 HBM 层），而尾部延迟可能*优于*生产环境（冷读取比混合命中/未命中流量更容易）。

**不要将这些数值引用为生产环境预测**，除非运行了具有现实层级配置的 v2 测试。
