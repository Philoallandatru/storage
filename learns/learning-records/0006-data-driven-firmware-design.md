# 从实验数据到 AI SSD 固件设计方法论

## Context
第六节课学习了如何将复测报告转化为 PPT 展示，并基于实验数据设计 AI SSD 固件方案。

## Key Insight
**数据驱动设计：每个设计决策必须有明确的证据链（证据 → 推理 → 设计 → 验证）**

这不是拍脑袋的创新，而是基于实测数据的工程推导。

## What I Learned

### 1. PPT 精炼方法：两页框架

**第 1 页：实验证据**
- 左半部分：性能提升（Cache hit +42%, TTFT -16%）
- 右半部分：I/O 模式量化（读写比 70:30，I/O 大小分布）
- 底部：关键发现（3 条）

**第 2 页：设计方案**
- 4 个证据链：读写比 → 非对称缓存，I/O 大小 → 粒度优化等
- 每个链路：证据 → 设计 → 验证指标
- 底部：核心验证指标

**启示**：
- 技术 PPT 不是堆砌数据，而是讲故事
- 故事线：问题 → 证据 → 解决方案 → 验证
- 2 页够了，每页一个核心观点

### 2. 证据链构建方法

**链路 1：读写比 70:30 → 非对称缓存**
```
证据：NVMe 读 4.58GB，写 1.94GB
推理：读主导，不是 50:50 均衡
设计：读缓存 8GB，写缓冲 200GB SLC，读优先 QoS
验证：fio rwmixread=70，测 P99 读延迟
```

**链路 2：I/O 256-512KB → 中粒度优化**
```
证据：45% 在 256-512KB，30% 在 512KB-1MB
推理：不是 4KB 小块，也不是 256MB 超大块
设计：FTL 粒度 256KB，预读窗口 1MB
验证：fio bssplit=256k/45:512k/30...
```

**方法论**：
1. 找到关键数据点（读写比、I/O 大小、QD 分布）
2. 与传统假设对比（50:50，4KB，高 QD）
3. 推导差异化需求
4. 设计针对性方案
5. 制定可测量的验证指标

### 3. 固件设计三大模块

**模块 1：非对称缓存管理**
- DRAM 读缓存：8GB（LRU + 预读）
- SLC 读缓存：500GB（热点数据）
- SLC 写缓冲：200GB（快速路径）
- TLC 主存储：9.5TB（冷数据）

**模块 2：QoS 调度器**
- 双队列：快速队列（QD<4）+ 批量队列（QD>=4）
- 优先级：读 > 写，快速 > 批量
- 避免 head-of-line blocking

**模块 3：热点检测**
- 触发：2 次访问 within 10min → 晋升 SLC
- 降级：30min 无访问 → 降级 TLC
- 目标：80% 热点命中率

**架构清晰度**：
每个模块都有明确的输入、输出和性能目标。

### 4. 实验配置的设计原则

**原则 1：直接验证单一变量**
- 配置 1：验证读写比 → rwmixread=70
- 配置 2：验证 I/O 大小 → bssplit=256k/45:512k/30...
- 配置 3：验证热点 → random_distribution=zipf:1.2
- 配置 4：验证 QD 混合 → 多 job 不同 iodepth

**原则 2：模拟真实场景**
- 使用 io_uring（Mooncake 使用的）
- direct=1（O_DIRECT，绕过页缓存）
- 运行时间足够（300-600s）

**原则 3：可测量的成功标准**
- 每个配置都有明确的验证指标
- P99 延迟 < 2ms（而不是"更快"）
- 命中率 > 80%（而不是"更高"）

### 5. 验证路线图：4 阶段 6 周

**阶段 1：基线测试**（1 周）
- 目标：确定改进空间
- 方法：商用 SSD + 4 个 fio 配置
- 输出：基线性能报告

**阶段 2：固件原型**（2 周）
- 目标：验证核心算法
- 方法：实现非对称缓存 + QoS
- 输出：算法有效性证明

**阶段 3：SLC/TLC 分层**（2 周）
- 目标：验证多层存储
- 方法：热点检测 + 晋升/降级
- 输出：命中率和延迟数据

**阶段 4：Mooncake 集成**（1 周）
- 目标：端到端验证
- 方法：运行原始实验
- 输出：性能提升对比

**关键点**：
- 逐步增量，每个阶段可独立验证
- 早期失败，快速调整
- 最后才做完整集成

## Why This Matters (For AI SSD预研)

### 可落地的设计方案

**不是空谈**：
- 每个设计都有明确的参数（8GB, 500GB, 200GB）
- 每个算法都有伪代码实现
- 每个指标都有验证方法

**可评估的成本**：
- DRAM：8GB × $10/GB = $80
- SLC：500GB × $0.5/GB = $250
- 总增量成本：~$330/盘（vs 性能提升）

**可复现的验证**：
- 4 个 fio 配置可直接运行
- 6 周路线图可直接执行
- 成功标准明确可测

### 数据驱动的说服力

**vs 拍脑袋方案**：
- "我觉得应该用 SLC" → "实测 P99 2.3ms，SLC 可降到 <1ms"
- "读缓存应该大一点" → "工作集 6.5GB，配 8GB 覆盖 100%"
- "需要热点检测" → "Zipf 分布，80% 访问集中在 20% 数据"

**说服链条**：
1. 展示实测数据（PPT 第 1 页）
2. 推导设计逻辑（PPT 第 2 页）
3. 提供验证方案（实验配置）
4. 预测性能提升（目标指标）

## Lessons for Future Work

### 做对的事

1. **先有数据，再做设计**
   - 不要基于假设设计
   - 用实测数据推导需求

2. **明确证据链**
   - 每个设计决策都要回答"为什么"
   - 证据 → 推理 → 设计，不能跳步

3. **可验证的方案**
   - 设计时就想好如何验证
   - 成功标准要可测量

4. **逐步增量**
   - 大方案拆成小阶段
   - 每个阶段独立可验证

### 避免的坑

1. **堆砌数据**
   - PPT 不是实验报告的复制粘贴
   - 提炼关键发现，讲清楚故事

2. **过度优化**
   - 不要优化不存在的问题
   - 聚焦实测瓶颈（SSD 读延迟）

3. **缺乏验证**
   - 设计再好，没验证就是空中楼阁
   - 先小规模验证，再大规模部署
