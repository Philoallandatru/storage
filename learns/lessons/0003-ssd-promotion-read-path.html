课程 3：SSD Promotion 读路径与零拷贝传输

🎯 学习目标
在本课结束时，你将能够：
  • 描述 SSD → Remote DRAM 的完整 promotion 流程（6 个步骤）
  • 理解三方协作模式：Requesting Client、Target Client、Master
  • 解释 ClientBuffer 的作用和生命周期管理
  • 识别读路径中的性能关键点
  • 对比 offload (写) 和 promotion (读) 的差异

触发场景

Promotion（提升）发生在以下情况：

⚠️ Cache Miss 场景
典型流程：
1. Decode worker 需要某个 KV Cache 对象
2. 在 GPU VRAM 中未找到 → 查询 Host DRAM
3. 在 Host DRAM 中未找到 → 查询 Remote DRAM Pool
4. 在 Remote DRAM 中未找到 → 查询 Master
5. Master 返回："这个对象在 SSD 上"
6. 触发 Promotion：SSD → Remote DRAM → Host DRAM → GPU VRAM

三方协作架构

与 offload 不同，promotion 涉及三个角色的协作：

┌─────────────────────────────────────────────────────────────┐ │ 角色 1: Requesting Client (请求者) │ │ • Decode worker 所在的节点 │ │ • 需要 KV Cache 数据 │ │ • 发起 Get 请求 │ └─────────────────────────────────────────────────────────────┘ │ │ ① BatchGet(keys) ↓ ┌─────────────────────────────────────────────────────────────┐ │ 角色 2: Master Service (协调者) │ │ • 维护对象元数据 │ │ • 知道数据在哪个节点的 SSD │ │ • 返回 LOCAL_DISK replica 信息 │ └─────────────────────────────────────────────────────────────┘ │ │ ② Replica info: {type: LOCAL_DISK, │ rpc_addr: "10.0.0.5:12345"} ↓ ┌─────────────────────────────────────────────────────────────┐ │ 角色 3: Target Client (SSD 拥有者) │ │ • 实际拥有 SSD 数据的节点 │ │ • 从 SSD 读取到 staging buffer │ │ • 通过 Transfer Engine 零拷贝传输 │ └─────────────────────────────────────────────────────────────┘

完整的 Promotion 流程

步骤 1：Cache Miss 检测

// Requesting Client (decode worker)
auto result = client_->Get(key, target_slices);

if (result.error() == ErrorCode::NOT_FOUND) {
    // 在内存中未找到，查询 Master
    auto replica_list = client_->BatchGet(keys);
}

步骤 2：Master 返回 SSD 副本信息

// Master Service 响应
QueryResult {
    replicas: [
        {
            type: ReplicaType::LOCAL_DISK,
            transport_endpoint: "10.0.0.5:12345",  // Target Client RPC 地址
            metadata: { ... }
        }
    ]
}
✅ 关键设计
Master 不参与数据传输！它只返回"去哪里找数据"，实际传输由 Client 之间直接完成。
这是典型的控制平面 / 数据平面分离设计。

步骤 3：RPC 请求 Target Client

Requesting Client 向 Target Client 发起 RPC：

// Requesting Client
auto response = target_client_rpc->batch_get_offload_object(
    keys,           // 需要的对象列表
    sizes,          // 每个对象的大小
    timeout_ms      // 超时时间
);

步骤 4：Target Client 从 SSD 读取

Target Client 收到请求后，执行以下操作：

// file_storage.cpp:723-793
// (a) 分配 ClientBuffer staging 空间
std::vector<std::string> single_key{storage_key};

auto staging = allocate_res.value();

// (b) 从 SSD 读取到 ClientBuffer

// BatchLoad 内部使用 BucketStorageBackend
// storage_backend.cpp:397
auto read_result = file->vector_read(
    aligned_buffer, aligned_size, aligned_offset);

// 使用 io_uring 批量读取
// uring_file.cpp:611
io_uring_prep_readv(sqe, fd, iov, iovcnt, offset);
io_uring_submit_and_wait(ring, batch);

ClientBuffer 的作用

🎨 ClientBuffer 架构

┌─────────────────────────────────────────────────────────────┐ │ ClientBuffer (Target Client 的 staging area) │ │ │ │ • 预先分配的内存区域（例如 20GB） │ │ • 预先注册到 Transfer Engine（一次注册，多次使用） │ │ • O_DIRECT 对齐（4KB） │ │ • 支持并发：多个 batch 可以同时占用不同的 slot │ │ │ │ ┌───────────────┬───────────────┬───────────────┐ │ │ │ Slot 1 │ Slot 2 │ Slot 3 │ │ │ │ (8GB) │ (8GB) │ (4GB) │ │ │ │ batch_id=101 │ batch_id=102 │ (free) │ │ │ └───────────────┴───────────────┴───────────────┘ │ │ │ │ 生命周期管理： │ │ • Allocate: 分配 slot，返回 batch_id │ │ • Use: Transfer Engine 通过 RDMA 读取数据 │ │ • Release: 显式释放，或 GC 超时回收 │ └─────────────────────────────────────────────────────────────┘

为什么需要 ClientBuffer？

  • 零拷贝传输：预注册到 Transfer Engine，RDMA 可以直接访问
  • O_DIRECT 对齐：满足 SSD 的 4KB 对齐要求
  • 并发支持：多个 promotion 请求可以并行处理
  • 生命周期隔离：每个 batch 有独立的 TTL，互不干扰

步骤 5：返回 Staging Buffer 地址

Target Client 返回响应给 Requesting Client：

BatchGetOffloadObjectResponse {
    batch_id: 101,                          // 用于后续释放
    pointers: [0x7f1234000000, ...],        // ClientBuffer 中的地址
    transfer_engine_addr: "10.0.0.5:9000",  // Transfer Engine 监听地址
    gc_ttl_ms: 30000                        // 30 秒后自动 GC
}
🤔 思考题
为什么需要 gc_ttl_ms？如果 Requesting Client 崩溃了会发生什么？
点击查看答案

容错机制：
如果 Requesting Client 在传输完成前崩溃：
• 它不会调用 release_offload_buffer()
• ClientBuffer 的 slot 会一直被占用
• GC 线程会在 30 秒后自动释放这个 slot
• 避免资源泄漏

这是一个经典的lease 机制：时间到期自动回收。

步骤 6：Transfer Engine 零拷贝传输

Requesting Client 使用 Transfer Engine 拉取数据：

// Requesting Client
auto transfer_result = client_->BatchGetOffloadObject(
    transfer_engine_addr,   // "10.0.0.5:9000"
    keys,                   // 对象列表
    pointers,               // ClientBuffer 中的地址
    target_slices           // 本地目标内存
);

// Transfer Engine 内部使用 RDMA
// rdma_transport.cpp
ibv_post_send(qp, &wr, &bad_wr);  // RDMA Read
// 从 Target Client 的 ClientBuffer 直接读取到 Requesting Client 的内存
// 零拷贝，无 CPU 参与

// 传输完成后立即释放 ClientBuffer
target_client_rpc->release_offload_buffer(batch_id);

步骤 7：提升到 Remote DRAM

数据传输到 Requesting Client 后，还需要在 Remote DRAM Pool 中创建副本：

// file_storage.cpp:770-793
// (a) 在 Remote DRAM Pool 中分配空间
auto alloc_result = client_->PromotionAllocStart(key, size, config);

// (b) 将 staging 数据写入 Remote DRAM
ErrorCode write_err = client_->PromotionWrite(
    alloc_result.value().memory_descriptor, tx_slices);

// (c) 通知 Master 提升成功
auto notify_res = client_->NotifyPromotionSuccess(key, tenant_id);
✅ 完成！
现在对象有两个副本：
• SSD 副本（保留，供未来可能的访问）
• Remote DRAM 副本（新创建，供后续快速访问）

后续对同一对象的访问会命中 Remote DRAM（50-100μs），而不是 SSD（1-5ms）。

性能分析

延迟分解

步骤 延迟 说明
Master 查询 ~100 μs RPC 往返，查询元数据
Target Client RPC ~50 μs 发起 batch_get_offload_object
SSD 读取 1-5 ms 主要瓶颈！
RDMA 传输 50-100 μs 零拷贝，取决于数据大小
Remote DRAM 写入 ~100 μs RDMA Write
总计 ~2-6 ms SSD 读取占主导
🎯 AI SSD 的优化目标
SSD 读取延迟（1-5ms）是整个 promotion 路径的主要瓶颈

对于 AI SSD 设计：
• P99 延迟比平均延迟更重要（避免长尾延迟）
• 预读策略：Bucket 聚合导致可能读取 256MB，但实际只需要其中几个对象
• 队列深度：io_uring 可能批量提交 32 个读请求，SSD 需要支持高 QD

带宽利用

FAST'25 论文中的实测数据（RAID0 配置）：

  • 顺序读取：~27 GB/s（4 × NVMe，接近饱和）
  • io_uring 批量：QD=32 时带宽最优
  • RDMA 传输：~87 GB/s（4 × 200 Gbps）

瓶颈分析：RDMA 带宽 > SSD 带宽，所以 SSD 是瓶颈。

Offload vs Promotion 对比

维度 Offload (写) Promotion (读)
触发方式 心跳轮询，批量 Cache miss，实时
延迟要求 低（后台异步） 高（阻塞请求）
数据流向 DRAM → SSD SSD → Remote DRAM
参与方 Target Client + Master Requesting Client + Target Client + Master
零拷贝 Host → SSD（DMA） SSD → Host（DMA）
Host → Remote（RDMA）
批处理 心跳周期内累积 单次请求，但 io_uring 批量
性能瓶颈 写带宽（通常不是瓶颈） 读延迟（P99 很重要）

ClientBuffer 生命周期管理

// file_storage.cpp:204-218
// 初始化：预注册到 Transfer Engine
void FileStorage::Init() {
    // 创建 ClientBuffer allocator
    client_buffer_allocator_ = std::make_shared<AlignedClientBufferAllocator>(
        config.local_buffer_size);
    
    // 注册到 io_uring（固定缓冲区优化）
    if (config.use_uring) {
        void* base_ptr = aligned_allocator->get_base_pointer();
        size_t size = aligned_allocator->get_total_size();
        UringFile::register_global_buffer(base_ptr, size);
    }
    
    // 注册到 Transfer Engine（RDMA 零拷贝）
    client_->RegisterLocalMemory(
        client_buffer_allocator_->getBase(),
        config.local_buffer_size,
        kWildcardLocation, false, true);
}

// 分配：为 promotion 请求分配 slot
auto staging = AllocateBatch(keys, sizes);
// staging->slices 包含 ClientBuffer 中的地址

// 使用：Transfer Engine 直接 RDMA 读取
client_->BatchGetOffloadObject(addr, keys, pointers, target_slices);

// 释放：显式释放或 GC 超时回收
release_offload_buffer(batch_id);

对 AI SSD 设计的启示

1. 读延迟是关键路径

观察：Promotion 是同步操作，阻塞 decode worker

AI SSD 应该：

  • 优化 P99 读延迟，而不仅仅是平均值
  • 减少延迟抖动（jitter）
  • QoS 保障：读请求优先级高于后台写入

2. 智能预读与 Bucket

观察：读取一个对象可能需要加载整个 256MB bucket

AI SSD 可以考虑：

  • 固件层预读：识别 bucket 访问模式
  • 部分读取优化：如果只需要 bucket 中的一个对象，能否跳过其他部分？
  • SSD 内缓存：热门 bucket 的元数据常驻

3. 高队列深度支持

观察：io_uring 批量提交最多 32 个读请求

AI SSD 应该：

  • 支持 QD=32 或更高而不降低单请求延迟
  • 内部并行：多个 die/channel 并行处理
  • 避免 head-of-line blocking

4. 与网络带宽匹配

观察：RDMA 带宽（87 GB/s）> SSD 带宽（27 GB/s）

AI SSD 目标：

  • 读带宽至少 30 GB/s（匹配 PCIe Gen4 x16）
  • 未来考虑 PCIe Gen5（~60 GB/s）
