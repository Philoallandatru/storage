课程 1：理解 Mooncake 的四层存储架构

🎯 学习目标
在本课结束时，你将能够：
  • 说出四层存储的名称、容量和延迟特征
  • 解释为什么需要多层存储架构
  • 识别每一层在 AI SSD 设计中的启示

为什么需要多层存储？

在 LLM 推理中，KV Cache 是性能的关键。它存储了 transformer 每一层的 attention keys 和 values，避免重复计算。但问题是：

⚠️ 核心挑战
对于长上下文 LLM，KV Cache 非常大

以 LLaMA3-70B 处理 128k tokens 为例：
• 单个请求的 KV Cache ≈ 40 GB
• 单张 H100 GPU (80GB VRAM) 只能容纳 2 个这样的请求
• 但生产环境需要同时服务数百个请求！

解决方案？分层存储——将热数据放在快速但昂贵的存储，将冷数据放在慢速但便宜的存储。

四层架构详解

Tier 1: GPU VRAM (最热层)

属性 值 AI SSD 启示
延迟 <1 μs GPU 计算是纳秒级，存储不能成为瓶颈
容量 40-80 GB/GPU 有限容量驱动了 offload 需求
管理者 推理引擎 (vLLM/SGLang) 应用层直接管理
用途 活跃 decode 的 KV Cache 最热的工作集
💡 思考题
如果一个 GPU 正在 decode 10 个请求，每个请求的 KV Cache 是 8GB，这 10 个请求能全部放在 GPU VRAM 吗？（假设 GPU 有 80GB VRAM）
点击查看答案

不能。 10 × 8GB = 80GB，看似刚好，但 GPU 还需要存储：
• 模型权重 (~140GB for 70B model，通过 tensor parallel 分片）
• 中间激活值
• 操作缓冲区
实际上，VRAM 的绝大部分被模型权重占用，留给 KV Cache 的空间有限。

Tier 2: Host DRAM (本地缓存)

属性 值 AI SSD 启示
延迟 ~10 μs PCIe 传输（cudaMemcpy）是主要开销
容量 100-500 GB/节点 比 GPU 大 5-10×，但仍不够
管理者 HiCache (SGLang) LRU/分层缓存策略
用途 最近使用的 KV Cache GPU 驱逐的第一站
✅ 关键观察
从 GPU 到 Host DRAM 的传输使用 cudaMemcpy，这是一次显式拷贝（不是零拷贝）。
带宽：PCIe Gen4 x16 = ~30 GB/s
对于 AI SSD 设计：如果能实现 GPU ↔ SSD 直接传输（GPUDirect Storage），可以跳过这一层。

Tier 3: Remote DRAM Pool (集群缓存)

属性 值 AI SSD 启示
延迟 50-100 μs RDMA 网络延迟，比 SSD 快 10-50×
容量 TB 级（弹性） 跨节点聚合容量
管理者 Mooncake Store 分布式元数据管理
用途 集群级 KV Cache 共享 多实例复用缓存

为什么有这一层？ 因为不同 GPU 节点可能处理相似的请求（例如相同的 system prompt）。通过 RDMA 共享 KV Cache，避免每个节点都重新计算。

Tier 4: Local NVMe SSD (冷存储) ⭐ 重点

属性 值 AI SSD 启示
延迟 1-5 ms 比 DRAM 慢 1000×，但比重新计算快 1000×
容量 多 TB/节点 成本效益的大容量存储
管理者 FileStorage 异步 offload/promotion
用途 冷 KV Cache 持久化 最后的防线
🎯 AI SSD 的核心价值
SSD 不是用来"快"的，而是用来"防止慢"的

考虑两个场景：
场景 A (无 SSD offload): Cache miss → 重新运行 prefill → 10-30 秒
场景 B (有 SSD offload): Cache miss → 从 SSD 加载 → 1-5 毫秒

SSD 的 1-5ms 延迟虽然比 DRAM 慢 1000×，但比重新计算快 10,000×！
这就是为什么 SSD offload 能让吞吐量提升 2.4× 的原因。

层间数据流动

数据不是随机跳跃的，而是遵循一个清晰的层级关系：

驱逐路径 (Eviction):
GPU VRAM → Host DRAM → Remote DRAM → SSD
         (cudaMemcpy)   (RDMA)      (io_uring)

提升路径 (Promotion):
SSD → Remote DRAM → Host DRAM → GPU VRAM
   (io_uring)     (RDMA)    (cudaMemcpy)

🧪 练习：识别传输方法

请填写每个层间传输使用的技术：

  1. GPU → Host: ___________
  2. Host → Remote DRAM: ___________
  3. Remote DRAM → SSD: ___________
查看答案
  1. cudaMemcpy (显式拷贝，~10 μs，30 GB/s)
  2. RDMA (零拷贝，50-100 μs，87 GB/s @ 4×200Gbps)
  3. io_uring + O_DIRECT (零拷贝，1-5 ms，27 GB/s)

真实性能数据

以下是 FAST'25 论文中的实测数据（多轮对话基准测试）：

指标 无 SSD Offload 有 SSD Offload 改善
Cache 命中率 (Round 7+) 36% 84% +133%
TTFT (首 token 时间) 16.0 秒 9.4 秒 -41%
吞吐量 1.0× (基准) 2.4× +140%

为什么提升这么大？ 因为在 Round 7+ 时，DRAM 已经耗尽。没有 SSD 的系统开始频繁驱逐 KV Cache，导致大量重新计算。有 SSD 的系统则能从 SSD 快速恢复，避免昂贵的重新计算。

对 AI SSD 设计的启示

🔍 设计考虑
基于四层架构，AI SSD 应该考虑：
  1. 顺序读优化：数据以大块（256MB buckets）批量读写
  2. 低延迟优先：1ms 和 5ms 的差异很重要（不像传统存储的 10ms vs 15ms）
  3. 高带宽：需要接近 30 GB/s 才能匹配 PCIe 带宽
  4. O_DIRECT 友好：绕过页缓存是必须的，确保 4KB 对齐性能良好
  5. 异步 I/O：offload 是后台操作，不能阻塞前台读取

小结

今天你学习了 Mooncake 的四层存储架构：

  • ✅ Tier 1 (GPU VRAM): <1μs, 40-80GB — 活跃计算
  • ✅ Tier 2 (Host DRAM): ~10μs, 100-500GB — 本地缓存
  • ✅ Tier 3 (Remote DRAM): 50-100μs, TB级 — 集群共享
  • ✅ Tier 4 (SSD): 1-5ms, 多TB — 冷存储

关键洞察：SSD 的价值不是"快"，而是"比重新计算快 10,000×"。
