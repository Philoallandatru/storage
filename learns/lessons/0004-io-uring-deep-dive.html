🎯 学习目标

为什么需要 io_uring？

⚠️ 传统 POSIX I/O 的问题
pread/pwrite 的性能瓶颈：
• 每次 I/O 都需要系统调用 (context switch)
• 阻塞式操作，无法充分利用 NVMe 并行性
• 批量操作需要多次系统调用
• 内核需要验证用户态缓冲区（页面固定）

结果：高 QPS 场景下，CPU 时间浪费在系统调用开销上

核心概念

组件 全称 作用
SQ Submission Queue 用户提交 I/O 请求的环形队列
CQ Completion Queue 内核返回完成事件的环形队列
SQE Submission Queue Entry 单个 I/O 请求描述符
CQE Completion Queue Entry 单个完成事件

关键优势

  • 共享内存：SQ/CQ 通过 mmap 共享，避免数据拷贝
  • 批量提交：一次 io_uring_submit() 可以提交多个 SQE
  • 批量收割：一次 io_uring_peek_batch_cqe() 收割多个 CQE
  • 真正异步：提交后立即返回，无需等待完成

Mooncake 的 io_uring 实现

线程本地环 (Thread-Local Ring)

// uring_file.cpp
thread_local std::unique_ptr g_thread_ring = nullptr;

static io_uring* GetThreadLocalRing() {
    if (!g_thread_ring) {
        g_thread_ring = std::make_unique();
        
        // 初始化环，深度 = 128
        int ret = io_uring_queue_init(128, g_thread_ring.get(), 0);
        if (ret < 0) {
            LOG(ERROR) << "io_uring_queue_init failed";
            g_thread_ring.reset();
            return nullptr;
        }
    }
    return g_thread_ring.get();
}
✅ 关键设计：每线程一个环
为什么不用全局环？

全局环的问题：
• 多线程竞争一个 SQ，需要加锁
• 锁竞争开销 > 1ms（高并发时）
• 限制了并行度

线程本地环的优势：
• 零锁竞争（每线程独立）
• 充分并行（多个环同时工作）
• 简化代码（无需同步）

批量提交流程

// uring_file.cpp:595-638
int UringFile::vector_write(const iovec* iovs, int iovcnt, off_t offset) {
    auto* ring = GetThreadLocalRing();
    if (!ring) {
        // Fallback to POSIX
        return posix_vector_write(iovs, iovcnt, offset);
    }
    
    // 分批处理，每批最多 QUEUE_DEPTH (32) 个请求
    for (int i = 0; i < iovcnt; i += QUEUE_DEPTH) {
        int batch = std::min(QUEUE_DEPTH, iovcnt - i);
        
        // (1) 准备 SQE
        for (int j = 0; j < batch; j++) {
            struct io_uring_sqe* sqe = io_uring_get_sqe(ring);
            if (!sqe) {
                LOG(ERROR) << "SQ is full";
                return -1;
            }
            
            // 准备写请求
            io_uring_prep_writev(sqe, fd_, &iovs[i+j], 1, offset);
            offset += iovs[i+j].iov_len;
        }
        
        // (2) 批量提交
        int submitted = io_uring_submit_and_wait(ring, batch);
        if (submitted < 0) {
            LOG(ERROR) << "io_uring_submit failed";
            return submitted;
        }
        
        // (3) 收割完成事件
        for (int j = 0; j < batch; j++) {
            struct io_uring_cqe* cqe;
            int ret = io_uring_wait_cqe(ring, &cqe);
            if (ret < 0) {
                LOG(ERROR) << "io_uring_wait_cqe failed";
                return ret;
            }
            
            // 检查错误
            if (cqe->res < 0) {
                LOG(ERROR) << "I/O error: " << strerror(-cqe->res);
            }
            
            io_uring_cqe_seen(ring, cqe);
        }
    }
    
    return 0;
}

固定缓冲区优化

问题：页面固定开销

每次 I/O 操作，内核需要：

  1. 验证用户态缓冲区地址合法
  2. 固定（pin）物理页面，防止换出
  3. 建立 DMA 映射
  4. I/O 完成后解除固定（unpin）

开销：每次 I/O ~100-200 ns

解决方案：预注册缓冲区

// file_storage.cpp:204-218
#ifdef USE_URING
if (config.use_uring) {

        std::static_pointer_cast(
            client_buffer_allocator_);

        size_t size = aligned_allocator->get_total_size();
        
        // 预注册到 io_uring
        if (UringFile::register_global_buffer(base_ptr, size)) {
            LOG(INFO) << "Successfully registered buffer: "
                      << "base=" << base_ptr << ", size=" << size;
        }
    }
}
#endif

实现细节

// uring_file.cpp
static std::atomic g_buf_base{nullptr};
static std::atomic g_buf_size{0};

bool UringFile::register_global_buffer(void* base, size_t size) {
    // (1) 禁用 Transparent Huge Pages
    // 原因：THP 会导致 FOLL_LONGTERM 页面固定失败
    madvise(base, size, MADV_NOHUGEPAGE);
    
    // (2) 保存全局变量（每线程环会独立注册）
    g_buf_base.store(base, std::memory_order_release);
    g_buf_size.store(size, std::memory_order_release);
    
    return true;
}

static void ensure_buf_registered(io_uring* ring) {
    void* base = g_buf_base.load(std::memory_order_acquire);
    size_t size = g_buf_size.load(std::memory_order_acquire);
    
    if (!base) return;
    
    // 注册到当前线程的环
    struct iovec iov = {base, size};
    int ret = io_uring_register_buffers(ring, &iov, 1);
    if (ret < 0) {
        LOG(WARNING) << "io_uring_register_buffers failed";
    }
}
💡 MADV_NOHUGEPAGE 的作用
问题：Linux THP (Transparent Huge Pages) 默认启用
• 使用 2MB 大页面
• io_uring 的 FOLL_LONGTERM 固定要求 4KB 小页面
• 冲突导致注册失败

解决：madvise(MADV_NOHUGEPAGE) 强制使用 4KB 页面

使用固定缓冲区

// 检查地址是否在注册区域内
bool is_in_registered_buffer(void* ptr) {
    void* base = g_buf_base.load();
    size_t size = g_buf_size.load();
    return ptr >= base && ptr < (char*)base + size;
}

// 准备读请求
if (is_in_registered_buffer(buffer)) {
    // 使用固定缓冲区 API（更快）
    io_uring_prep_read_fixed(sqe, fd, buffer, len, offset, buf_index);
} else {
    // 普通 API
    io_uring_prep_read(sqe, fd, buffer, len, offset);
}

O_DIRECT 与对齐

为什么使用 O_DIRECT？

⚠️ 页缓存的问题
Linux 页缓存（Page Cache）虽然快，但有缺点：
• 延迟不可预测（缓存命中 vs miss）
• 占用宝贵的 DRAM
• 重复缓存（应用层 + OS 层）
• 污染其他应用的缓存

对于 Mooncake：
• 已有多层缓存（GPU/Host/Remote DRAM）
• OS 页缓存是冗余的
• 需要确定性延迟

对齐要求

项目 要求 说明
缓冲区地址 4KB 对齐 必须是 4096 的倍数
I/O 大小 4KB 对齐 读写字节数是 4096 的倍数
文件偏移 4KB 对齐 offset 是 4096 的倍数

对齐辅助函数

// storage_backend.cpp
static constexpr size_t kDirectIOAlignment = 4096;

static inline size_t align_up(size_t size, size_t alignment) {
    return (size + alignment - 1) & ~(alignment - 1);
}

static inline int64_t align_down(int64_t offset, int64_t alignment) {
    return offset & ~(alignment - 1);
}

// 读取时的对齐处理
int64_t aligned_offset = align_down(offset, kDirectIOAlignment);
size_t aligned_size = align_up(offset - aligned_offset + size, 
                               kDirectIOAlignment);
void* aligned_buffer = allocate_aligned(aligned_size);

// 实际读取
file->read_aligned(aligned_buffer, aligned_size, aligned_offset);

// 拷贝到用户缓冲区（只拷贝需要的部分）
memcpy(user_buffer, aligned_buffer + (offset - aligned_offset), size);

对 AI SSD 的启示

1. 支持高队列深度

观察：io_uring 批量提交 32 个请求

AI SSD 应该：

  • 内部并行处理能力：至少 32 并发
  • 避免 head-of-line blocking
  • QD=32 时，单个请求延迟不应显著增加

2. O_DIRECT 性能优化

观察：所有生产 I/O 使用 O_DIRECT

AI SSD 应该：

  • DMA 引擎高效处理 4KB 对齐的传输
  • 无额外的固件层缓冲开销
  • 确保 O_DIRECT 路径是最快路径

3. 批量操作优化

观察：批量提交带来 3× 性能提升

AI SSD 应该：

  • 识别批量操作并优化调度
  • 内部合并相邻请求
  • 批量完成通知
