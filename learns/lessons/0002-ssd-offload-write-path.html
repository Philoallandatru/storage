课程 2：SSD Offload 写路径深度解析

🎯 学习目标
在本课结束时，你将能够：
  • 描述从内存到 SSD 的完整数据流程（5 个步骤）
  • 解释为什么 Mooncake 不使用 GPUDirect Storage
  • 理解 GPU 指针检测和暂存机制
  • 识别 offload 路径中的拷贝边界和零拷贝边界
  • 为 AI SSD 提出至少 2 个写路径优化建议

核心问题

当 Remote DRAM Pool 的内存不足时，Master Service 需要将一些冷 KV Cache 对象驱逐到 SSD。这个过程被称为 Offload。

⚠️ 设计挑战
数据可能在三个不同的位置：
1. GPU VRAM（如果刚从 decode worker 驱逐）
2. Host DRAM（本地缓存）
3. Remote DRAM（已经在远程节点）

FileStorage 必须能够处理所有三种情况，并高效地写入 SSD。

完整的 Offload 流程

步骤 1：心跳触发

Offload 不是由应用程序直接触发的，而是由 Heartbeat Thread 在后台定期执行。

// file_storage.cpp:582-684
tl::expected<void, ErrorCode> FileStorage::Heartbeat() {
    // 向 Master 询问：有哪些对象需要 offload？
    std::vector<OffloadTaskItem> offloading_objects;
    auto heartbeat_result = client_->OffloadObjectHeartbeat(
        enable_offloading_, offloading_objects);
    
    // Master 返回需要 offload 的对象列表
    if (heartbeat_result && !offloading_objects.empty()) {
        auto offload_result = OffloadObjects(offloading_objects);
    }
    
    return tl::expected<void, ErrorCode>();
}
🤔 思考题
为什么使用心跳（定期轮询）而不是事件驱动（Master 主动推送）？
点击查看答案

三个原因：
1. 解耦：Master 不需要维护到每个 Client 的反向连接
2. 批处理：可以在一个心跳周期内累积多个 offload 任务，批量处理
3. 容错：如果 Client 暂时不可达，Master 只需等待下次心跳

默认心跳间隔：通过 MOONCAKE_OFFLOAD_HEARTBEAT_INTERVAL_SECONDS 配置

步骤 3：GPU 指针检测与暂存 ⭐ 关键

这是 Mooncake SSD offload 最有趣的部分：自动检测并处理 GPU 指针。

// file_storage.cpp:500-518
for (auto& [obj_key, slices] : batch_object) {
    std::vector<Slice> host_slices;
    std::vector<PinnedBuffer> staging_bufs;
    
    for (const auto& slice : slices) {
        int device_id = -1;
        
        // 🔍 检测：这是 GPU 指针吗？
        if (IsDevicePointer(slice.ptr, &device_id)) {
            SetDevice(device_id);
            
            // 📦 分配暂存缓冲区（Host pinned memory）
            auto buf = pinned_buffer_pool_->Acquire(slice.size);
            
            // 📋 显式拷贝：GPU → Host
            if (!CopyDeviceToHost(buf.data, slice.ptr, slice.size)) {
                LOG(ERROR) << "D2H staging failed for key: " << obj_key;
                obj_success = false;
                break;
            }
            
            host_slices.emplace_back(Slice{buf.data, slice.size});
            staging_bufs.push_back(buf);  // 保持 RAII 生命周期
        } else {
            // ✅ 已经在 Host，直接使用
            host_slices.push_back(slice);
        }
    }
}
🚨 关键发现：不使用 GPUDirect Storage
Mooncake 在生产环境中不使用 GDS 进行本地 SSD offload

证据：
• 使用 cudaMemcpy，不是 cuFile* API
• 所有数据必须先到达 Host DRAM
• GDS 代码仅存在于 NVMe-oF transport（远程 NVMe，不是本地 SSD）

为什么？
1. 批处理语义：FileStorage 处理来自不同 tenants/segments 的对象批次，GDS 优化的是大块连续传输
2. 数据源通常已在 DRAM：大多数情况下，数据已经从远程节点驱逐到本地 DRAM，GPU→SSD 直接路径是罕见情况
3. 复杂度 vs 收益：GDS 需要 CUDA context 管理、设备指针跨进程追踪、cuFile 驱动开销

GPU 指针检测实现

// gpu_staging_utils.h:17-45
inline bool IsDevicePointer(const void* ptr, int* out_device_id) {
#if defined(USE_CUDA)
    cudaPointerAttributes attr{};
    
    // CUDA 运行时查询指针属性
    if (cudaPointerGetAttributes(&attr, ptr) == cudaSuccess &&
        attr.type == cudaMemoryTypeDevice) {
        
        if (out_device_id) *out_device_id = attr.device;
        return true;  // ✅ 这是 GPU 指针
    }
#endif
    return false;  // ❌ 这是 Host 指针
}

inline bool CopyDeviceToHost(void* host_dst, const void* device_src, 
                             size_t size) {
#if defined(USE_CUDA)
    // 同步拷贝：GPU → Host
    return cudaMemcpy(host_dst, device_src, size, 
                     cudaMemcpyDeviceToHost) == cudaSuccess;
#else
    return false;
#endif
}

io_uring vs POSIX I/O

特性 io_uring (现代) pwritev (传统)
系统调用 批量提交，单次 io_uring_submit() 每次 I/O 一次 pwritev()
异步 真正异步，提交后立即返回 阻塞直到完成
批处理 最多 32 个请求/批次 需要多次调用
CPU 开销 低（共享内存环） 高（每次 context switch） 
// uring_file.cpp:595-638
int UringFile::vector_write(const iovec* iovs, int iovcnt, off_t offset) {
    auto* ring = GetThreadLocalRing();  // 每线程一个环，无锁
    
    // 准备 SQE (Submission Queue Entry)
    for (int i = 0; i < iovcnt; i += QUEUE_DEPTH) {
        int batch = std::min(QUEUE_DEPTH, iovcnt - i);
        
        for (int j = 0; j < batch; j++) {
            auto* sqe = io_uring_get_sqe(ring);
            io_uring_prep_writev(sqe, fd_, &iovs[i+j], 1, offset);
            offset += iovs[i+j].iov_len;
        }
        
        // 批量提交
        io_uring_submit_and_wait(ring, batch);
        
        // 收集完成事件
        for (int j = 0; j < batch; j++) {
            io_uring_cqe* cqe;
            io_uring_wait_cqe(ring, &cqe);
            // ... 检查错误 ...
            io_uring_cqe_seen(ring, cqe);
        }
    }
}

拷贝边界总结

传输路径 零拷贝？ 延迟 带宽
GPU → Host ❌ 否（cudaMemcpy） ~10 μs ~30 GB/s
Host → SSD ✅ 是（O_DIRECT DMA） 1-5 ms ~27 GB/s
Remote → Host ✅ 是（RDMA） 50-100 μs ~87 GB/s

BucketStorageBackend 设计

为什么使用 Bucket？

✅ 聚合小对象
问题：如果每个 KV Cache 对象一个文件，100 万个对象 = 100 万个文件
• 文件系统元数据开销巨大
• 打开/关闭文件的系统调用开销
• 碎片化严重

解决方案：将多个对象打包到一个 bucket 文件
• 默认 256MB 或 500 个对象/bucket
• 两个文件：.bucket（数据）+ .meta（索引）
• 大大减少文件数量和元数据开销

Bucket 文件格式

bucket_12345678.bucket:
┌─────────────────────────────────────────────────┐
│  Object 1 (key="sess_001", size=8MB)            │
├─────────────────────────────────────────────────┤
│  Object 2 (key="sess_002", size=12MB)           │
├─────────────────────────────────────────────────┤
│  ...                                            │
├─────────────────────────────────────────────────┤
│  Object N (key="sess_500", size=6MB)            │
└─────────────────────────────────────────────────┘
Total: 256MB

bucket_12345678.meta:
{
  "bucket_id": 12345678,
  "keys": ["sess_001", "sess_002", ..., "sess_500"],
  "offsets": [0, 8388608, 20971520, ...],
  "sizes": [8388608, 12582912, ..., 6291456],
  "total_size": 268435456
}

Bucket 大小的权衡

Bucket 大小 优点 缺点
64MB（小） • 读取延迟低
• 驱逐粒度小		 • 文件数量多
• 元数据开销大
256MB（默认） • 平衡点
• 文件数量合理		 • 读取可能拉取无关数据
1GB（大） • 最少文件数
• 顺序 I/O 友好		 • 读取延迟高
• 驱逐不灵活
🔧 配置参数
Bucket 大小可通过环境变量配置：
MOONCAKE_OFFLOAD_BUCKET_SIZE_LIMIT=268435456 (256MB)
MOONCAKE_OFFLOAD_BUCKET_KEYS_LIMIT=500 (最多 500 个对象)

对于 AI SSD：256MB 是否是最优值？考虑 NVMe 的预读特性和队列深度。

对 AI SSD 设计的启示

💡 写路径优化方向

1. 大块顺序写入优化

观察：Bucket 聚合机制导致写入模式是大块顺序的（256MB）

AI SSD 应该：

  • 针对 64MB-512MB 范围的大块写入优化写缓冲策略
  • 识别顺序写入流，提前分配连续物理块
  • 减少写放大（Write Amplification），因为数据已经是大块的

2. O_DIRECT 性能

观察：所有生产 I/O 使用 O_DIRECT 绕过页缓存

AI SSD 应该：

  • 确保 O_DIRECT 路径没有额外的固件层缓冲开销
  • 支持 4KB 对齐的高效 DMA
  • 在固件层实现预读（而不是依赖 OS 页缓存）

3. 异步写入吞吐

观察：Offload 是后台异步操作，不阻塞前台读取

AI SSD 应该：

  • 提供大容量的写缓冲（DRAM + SLC cache）
  • QoS 机制：确保后台写入不影响前台读延迟
  • 支持高并发：io_uring 可能同时提交 32 个写请求

4. 未来：GPU → SSD 直接路径

观察：当前使用 cudaMemcpy 暂存，有拷贝开销

AI SSD 可以考虑：

  • 支持 GPUDirect Storage（需要 NVIDIA 认证）
  • 提供 cuFile-compatible 驱动
  • 但要评估：GPU-resident 数据比例是否高到值得这个复杂度

Pinned memory 的作用：
• cudaHostAlloc() 分配的页锁定内存
• 不会被 OS 换出到磁盘
• GPU 可以通过 DMA 直接访问（无需 staging）
• cudaMemcpy 性能更高（~30 GB/s vs ~10 GB/s）

小结

今天你深入学习了 SSD Offload 写路径：

  • ✅ 5 步流程：心跳触发 → 查询位置 → GPU 暂存 → SSD 写入 → 通知 Master
  • ✅ 关键发现：不使用 GPUDirect Storage，通过 cudaMemcpy 暂存
  • ✅ 零拷贝边界：Host → SSD 是零拷贝，GPU → Host 不是
  • ✅ Bucket 聚合：256MB 批量写入，减少文件数量
  • ✅ io_uring 优化：批量异步 I/O，每线程一个环

对 AI SSD 的核心启示：优化大块顺序写、O_DIRECT 性能、异步写缓冲
