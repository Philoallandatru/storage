工作负载特征的重要性

⚠️ 为什么不能靠猜测？
通用 SSD 的假设：
• 对象大小：4KB-64KB（数据库、文件系统）
• 访问模式：随机读写混合
• 生命周期：长期持久化

AI 工作负载的现实：
• 对象大小：几 MB 到几十 GB（KV Cache）
• 访问模式：顺序大块读、异步批量写
• 生命周期：短暂（会话结束即删除）

如果 AI SSD 按通用 SSD 设计 → 性能和寿命都会受损！

Kimi 真实 Trace 数据

FAST'25 论文公开了 Kimi（Moonshot AI）的生产环境 trace 数据。

数据集概览

维度 数值
时间跨度 7 天（2024年某周）
总请求数 ~280 万次会话
模型 Kimi-128k (类似 LLaMA3-70B)
KV Cache 总量 ~500 TB（峰值）
SSD Offload 占比 ~40%（热数据在 DRAM）

关键指标

  • 平均会话长度：7.2 轮对话
  • 平均序列长度：18k tokens（输入 + 输出）
  • Cache 命中率：84%（有 SSD）vs 36%（无 SSD）

对象大小分布

理论计算

对于 LLaMA3-70B 模型（80 层，8192 隐藏维度）：

KV Cache 大小 = 层数 × 2 (K+V) × 序列长度 × 隐藏维度 × 字节数

单个 token 的 KV Cache：
80 层 × 2 × 8192 × 2 字节 (FP16) = 2.56 MB / token

128k context 的完整 KV Cache：
2.56 MB/token × 131,072 tokens = 335 GB
💡 实际场景
多轮对话的增量特性：

第 1 轮：用户输入 1k tokens → KV Cache = 2.56 GB
第 2 轮：新增 500 tokens → 增量 KV Cache = 1.28 GB
第 3 轮：新增 800 tokens → 增量 KV Cache = 2.05 GB
...

SSD 存储的是完整或增量？ 完整！每轮更新后存储整个会话的 KV Cache。

分布特征（FAST'25 数据）

对象大小 占比 说明
< 1 GB 45% 短会话（1-3 轮）
1-5 GB 35% 中等会话（4-7 轮）
5-20 GB 15% 长会话（8-15 轮）
> 20 GB 5% 超长会话（16+ 轮或长上下文）
对象大小分布（简化示意）： 45% │ ████████████████████████████ │ 35% │ ████████████████████ │ 15% │ ████████ │ 5% │ ██ └──────────────────────────────── <1GB 1-5GB 5-20GB >20GB 中位数：~2 GB P90：~8 GB P99：~25 GB

对 AI SSD 的启示

  1. 优化目标大小：1-5 GB（70% 的对象）
  2. 支持大对象：需要处理 20+ GB 的单个对象
  3. 变长对象：不能假设固定大小

读写比例与访问模式

操作类型分布

操作 占比 特征
Offload (写) 20% 批量异步，顺序写入
Promotion (读) 70% 实时同步，顺序读取
Remove (删除) 10% 会话结束时批量删除
✅ 关键发现
读多写少（70:20）

这与传统数据库（50:50）或文件系统（30:70）完全不同！
原因：Cache 命中率 84%，大部分访问在 DRAM。只有 miss 才读 SSD。

访问模式分析

Offload 模式（写入）： ┌────────────────────────────────────────┐ │ 时间轴： │ │ ────▶ │ │ │ │ [对象A: 2GB] ──┐ │ │ [对象B: 1GB] ──┤ 批量写入 │ │ [对象C: 3GB] ──┤ (心跳触发) │ │ [对象D: 1.5GB]─┘ │ │ │ │ 特征： │ │ • 批量聚合（心跳周期：10-30秒） │ │ • 顺序写入同一 bucket │ │ • 异步后台操作 │ └────────────────────────────────────────┘ Promotion 模式（读取）： ┌────────────────────────────────────────┐ │ 时间轴： │ │ ────▶ │ │ │ │ Cache Miss → 读取对象X (3GB) │ │ ↓ │ │ 从 SSD 读取整个 bucket (256MB×N) │ │ ↓ │ │ 提取对象X │ │ ↓ │ │ 写入 Remote DRAM │ │ │ │ 特征： │ │ • 实时同步（阻塞 decode worker） │ │ • 可能读取整个 bucket │ │ • 对延迟敏感（P99 < 5ms） │ └────────────────────────────────────────┘

时间局部性与热点分析

会话生命周期

阶段 平均时长 SSD 状态
活跃期 5-15 分钟 频繁 offload/promotion
空闲期 数小时到数天 冷数据，很少访问
结束 - 删除 SSD 上的对象

热点数据识别

⚠️ Zipf 分布
20% 的会话产生 80% 的访问

热点特征：
• 系统提示词（system prompt）被多个会话复用
• 长上下文会话的多轮对话
• RAG 场景的知识库检索结果

这些热点数据应该优先保留在 DRAM 或 SSD 缓存中。

重访问间隔（Reuse Distance）

重访问间隔分布： 50% │ ████████████████████████████ │ ◄─ <1 分钟（多轮对话） 30% │ ████████████████ │ ◄─ 1-10 分钟（会话内） 15% │ ████████ │ ◄─ 10-60 分钟 5% │ ██ │ ◄─ >1 小时（很少） └──────────────────────────────── <1min 1-10 10-60 >60min 结论： • 50% 的重访问在 1 分钟内 • 80% 的重访问在 10 分钟内 • 超过 1 小时的重访问极少（冷数据）

对 AI SSD 的启示

  1. 短期热点缓存：SSD 内置 DRAM 缓存（用于 1-10 分钟窗口）
  2. 快速冷却：超过 1 小时未访问的数据可以降低优先级
  3. 批量删除优化：会话结束时集中删除，减少碎片

I/O 模式与队列深度

队列深度分布

QD 范围 占比 场景
QD = 1-4 40% 单个 promotion 请求
QD = 8-16 35% 小批量 offload
QD = 16-32 20% 大批量 offload
QD > 32 5% 峰值时刻
✅ 设计建议
优化 QD=1-32 范围

• QD=1-4：低延迟路径（用于 promotion）
• QD=16-32：高吞吐路径（用于 offload）
• QD>32：偶发峰值，不需过度优化

I/O 大小分布

由于 BucketStorageBackend 的聚合，实际 I/O 大小与对象大小不同：

I/O 大小 占比 说明
256 MB 60% 完整 bucket 读写
64-256 MB 25% 部分填充的 bucket
< 64 MB 15% 元数据或小对象

结论：AI SSD 应该针对 64-256 MB 的大块顺序 I/O 优化。

SSD 寿命与写放大

总写入量估算

假设场景：
• 100 万会话/天
• 平均 KV Cache 大小：2 GB/会话
• SSD offload 占比：40%

每天 SSD 写入量：
1,000,000 会话 × 2 GB × 40% = 800 TB/天

年写入量：
800 TB/天 × 365 天 = 292 PB/年

对于 10 TB SSD：
DWPD (Drive Writes Per Day) = 800 TB / 10 TB = 80 DWPD
🚨 超高 DWPD 需求
80 DWPD 是极高的写入负载！

对比：
• 企业级 SSD：1-3 DWPD
• 高性能 SSD：5-10 DWPD
• AI SSD 需要：50-100 DWPD

解决方向：
1. 使用 3D NAND 高耐久度芯片
2. 大容量 OP (Over-Provisioning)：30-50%
3. 智能 GC：识别冷数据，减少迁移
4. 生命周期感知：短暂数据写入 SLC

写放大因子（WAF）

Mooncake 的设计有助于降低 WAF：

特性 对 WAF 的影响
大块顺序写入 减少碎片，降低 GC 频率
批量删除 整 bucket 失效，无需迁移
短生命周期 数据很快过期，减少 GC 迁移

预期 WAF：2-3（vs 传统随机写的 10-20）

综合分析与设计建议

🎯 AI SSD 核心需求

1. 性能需求

指标 目标值 依据
顺序读带宽 > 30 GB/s 匹配 PCIe Gen4 x16
顺序写带宽 > 15 GB/s 后台 offload，要求较低
P99 读延迟 < 2 ms promotion 实时性
QD=32 延迟 < 3 ms 批量 I/O 常见场景

2. 容量需求

  • 单盘容量：10-30 TB（存储数千会话）
  • DRAM 缓存：64-128 GB（热点缓存 + 元数据）
  • OP 比例：30-50%（应对高 DWPD）

3. 耐久度需求

  • DWPD：50-100（5 年寿命）
  • 总写入量：100+ PB（10 TB 盘）
  • 目标 WAF：< 3

4. 特殊优化

  • 大对象支持：单个对象 20+ GB
  • 变长 I/O：64-256 MB 范围
  • 批量删除：快速释放空间
  • 短期热点缓存：1-10 分钟窗口

小结

今天你深入分析了 LLM 推理的工作负载特征：

  • ✅ 对象大小：1-5 GB 占主体（70%），支持 20+ GB
  • ✅ 读写比例：70:20（读多写少）
  • ✅ 访问模式：顺序大块（64-256 MB I/O）
  • ✅ 时间局部性：80% 重访问在 10 分钟内
  • ✅ DWPD 需求：50-100（超高写入负载）

核心发现：AI 工作负载与传统存储完全不同，需要专门设计的 AI SSD。
