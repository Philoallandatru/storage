逻辑链路：证据 → 推理 → 设计

链路 1：读多写少 → 非对称架构

步骤 内容
证据 NVMe 读 4.58GB，写 1.94GB → 读写比 70:30
推理 传统 SSD 假设读写均衡（50:50），但 AI 工作负载读主导
设计 • 增大读缓存（8GB DRAM）
• 减少写缓存但保证足够 SLC (200GB)
• QoS: 读请求抢占写请求
验证 fio 测试：70% 读 + 30% 写混合，测量 P99 读延迟

链路 2：中等 I/O 块 → 优化粒度

步骤 内容
证据 I/O 大小分布：256-512KB (45%), 512KB-1MB (30%)
推理 不是 4KB 小块（数据库），也不是 256MB 超大块
→ 中等粒度访问模式
设计 • FTL 映射粒度：256KB（vs 传统 4KB）
• 预读窗口：1MB（覆盖 75% 的 I/O）
• DMA 引擎：针对 512KB 优化传输
验证 fio 测试：bs=256k-1m 随机分布，测量吞吐量和延迟

链路 3：P99 延迟敏感 → 多层存储

步骤 内容
证据 实测 P99 读延迟 2.3ms，但 Mooncake 期望 < 2ms
推理 TLC 随机读：~2ms，SLC 随机读：~500μs
→ 需要热点识别机制
设计 • 2 层缓存：SLC (500GB) + TLC (9.5TB)
• 热点检测：LRU + 访问频率
• 晋升阈值：2 次访问 within 10 分钟 → SLC
• 降级：30 分钟无访问 → TLC
验证 模拟 Zipf 分布访问，测量热点命中率和 P99 延迟

链路 4：低队列深度 → 延迟优先调度

步骤 内容
证据 QD=1-8 占 75%，QD=8-32 占 25%
推理 大部分请求是低并发（单个 promotion）
→ 低延迟比高吞吐更重要
设计 • 双队列架构：
- 快速队列（QD<4）：优先调度，直接访问 SLC
- 批量队列（QD>=4）：延迟容忍，可访问 TLC
• 避免批量请求阻塞低延迟请求
验证 fio 混合：iodepth=1-4 (70%) + iodepth=16 (30%)
测量低 QD 请求的 P99 延迟

固件设计详细方案

模块 1：非对称缓存管理

读缓存层次结构： DRAM 读缓存 (8GB) │ ├─ LRU 缓存：最近读取的数据 │ └─ 命中率目标：> 30% │ └─ 预读缓冲：1MB 窗口 └─ 命中率目标：> 50% SLC 读缓存 (500GB) │ ├─ 热点数据：2 次访问 within 10min │ └─ 保留时间：30min 无访问 │ └─ 晋升队列：候选热点 └─ 降级队列：冷却数据 写缓冲层次： SLC 写缓冲 (200GB) │ ├─ 快速路径：QD < 4 │ └─ 目标延迟：< 500μs │ └─ 批量路径：QD >= 4 └─ 后台刷新到 TLC

模块 2：QoS 调度算法

伪代码：

class QoSScheduler {
    Queue fast_queue;    // QD < 4, 延迟敏感
    Queue batch_queue;   // QD >= 4, 吞吐优先
    
    Request schedule_next() {
        // 优先级规则
        if (!fast_queue.empty()) {
            // 读请求 > 写请求
            Request read_req = fast_queue.peek_read();
            if (read_req && read_req.age() > 1ms) {
                return fast_queue.pop();  // 紧急读
            }
        }
        
        // 批量队列：仅在快速队列空闲时处理
        if (fast_queue.empty() && !batch_queue.empty()) {
            return batch_queue.pop();
        }
        
        // 默认返回快速队列
        return fast_queue.pop();
    }
    
    void submit(Request req) {
        if (req.queue_depth < 4 || req.is_read()) {
            fast_queue.push(req);
        } else {
            batch_queue.push(req);
        }
    }
};

模块 3：热点检测与晋升

伪代码：

class HotspotDetector {
    HashMap access_map;
    const int HOTSPOT_THRESHOLD = 2;
    const Duration WINDOW = 10min;
    
    void on_read(LBA lba) {
        auto& record = access_map[lba];
        record.access_count++;
        record.last_access = now();
        
        // 晋升到 SLC
        if (record.access_count >= HOTSPOT_THRESHOLD &&
            (now() - record.first_access) < WINDOW) {
            promote_to_slc(lba);
            record.in_slc = true;
        }
    }
    
    void gc_cycle() {
        // 降级冷数据
        for (auto& [lba, record] : access_map) {
            if (record.in_slc &&
                (now() - record.last_access) > 30min) {
                demote_to_tlc(lba);
                record.in_slc = false;
            }
        }
    }
};

常见问题

❓ Q1: 为什么选择 8GB DRAM 缓存？
证据：实验中工作集约 6.5GB（4.58GB 读 + 1.94GB 写）
推理：8GB 可覆盖完整工作集 + 预读缓冲
权衡：更大的 DRAM 成本高，但命中率提升有限（边际递减）
❓ Q2: SLC 500GB 够用吗？
证据：Zipf 分布下，20% 数据产生 80% 访问
计算：10TB SSD × 20% = 2TB，但实际热点更集中
推理：500GB 可容纳真正的热点（top 5%），命中率 > 80%
❓ Q3: 如何应对 DWPD 需求？
实验数据：写入 1.94GB/run，假设 100 runs/天 = 194GB/天
计算：194GB/10TB = ~0.02 DWPD（远低于之前估算的 80 DWPD）
推理：本地测试环境写入量少，生产环境需要 × 100-500
设计：预留 200GB SLC 写缓冲 + 30% OP，支持 10-30 DWPD
